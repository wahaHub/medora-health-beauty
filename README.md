# Medora Health Beauty

Medora Health Beauty is the patient-facing beauty and medical tourism web app. The main app is a Vite + React project with a small local/admin API surface, Supabase-backed data, R2-hosted media assets, and a curated archive for legacy material that should not be part of the active runtime.

## Active Project Structure

```text
.
├── admin/                 # Local/admin Express server and admin public assets
├── api/                   # Serverless API routes used by the frontend and patient flows
├── docs/                  # Project documentation, audits, brand/reference/data files
├── migrations/            # Current Supabase schema snapshots and migration notes
├── public/                # Static assets served by Vite
├── scripts/               # Valid project scripts, including Supabase snapshot tooling
├── src/                   # Active frontend source code
├── test/                  # Vitest and Testing Library tests
├── index.html             # Vite HTML entry
├── package.json           # NPM scripts and dependency manifest
├── tailwind.config.js     # Tailwind build configuration
├── tsconfig.json          # TypeScript configuration
├── vite.config.ts         # Vite configuration
└── vitest.config.ts       # Test configuration
```

## Data And Migrations

- `migrations/000_supabase_current_schema_snapshot.sql` is the current Supabase `public` schema dump generated from the live project.
- `migrations/000_supabase_public_schema_openapi_snapshot.sql` is a lighter OpenAPI-derived snapshot kept for comparison and fallback context.
- Legacy hand-written migrations and temp seed files live under `archive/migrations-legacy/`.
- `npm run db:snapshot` loads `.env` and writes a fresh Supabase schema snapshot using `scripts/dump-supabase-schema.sh`.
- `npm run db:snapshot:openapi` regenerates the OpenAPI-derived snapshot without direct Postgres schema metadata.

## Environment Files

Runtime secrets must stay local. The repo ignores `.env`, `.env.*`, `*.env`, and `*.env.*`, while allowing template files such as `.env.example`, `.env.template`, and `.env.sample`.

The local `.env` is expected to provide:

- Supabase project URL and anon/service keys
- `SUPABASE_DB_URL` for schema snapshots
- R2 credentials and public media URL settings
- Admin console credentials/session secret
- Optional AI/API keys used by the app

Do not commit real credentials, database URLs, service-role keys, admin passwords, local Vercel env files, or archived external project env files.

## Archives

`archive/` is split by intent:

- `archive/infrastructure-cloudflare-legacy/` keeps the old Terraform deployment material for reference.
- `archive/migrations-legacy/` keeps superseded manual migrations and temp seed scripts.
- `archive/scripts-legacy/` keeps old scripts that are not part of the active workflow.
- `archive/external-projects/`, `archive/runtime-artifacts/`, and `archive/tooling/` are local-only buckets ignored by git because they can contain separate repos, generated output, node_modules, and secrets.

The old full `medical-crm-v2` project is not part of the active Medora Health Beauty runtime. Its local archived copy, if present, belongs under `archive/external-projects/` and should remain ignored.

## Notes

- `dist/` is generated by `npm run build` and is ignored.
- `scripts/data/` is generated/supporting data and is ignored because the canonical data lives in Supabase.
- External worktrees and agent/tool directories are ignored to keep this repository focused on the active project.
