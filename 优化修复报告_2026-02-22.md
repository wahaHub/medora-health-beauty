### **2. hospital-info 降级逻辑不完整** 🔴
**问题**:
- Supabase 返回 `{ data, error }`，不会 throw
- 之前只捕获 `Promise.allSettled` 的 `rejected`（网络级异常）
- 捕获不到 Supabase 返回的 `error` 字段（查询失败）
- 导致"静默缺数据、没 warnings、难排障"

**修复**:
```typescript
// 之前：只检查 rejected
const translation = translationResult.status === 'fulfilled' ? translationResult.value.data : null
if (translationResult.status === 'rejected') {
  warnings.push('Translation data unavailable')
}

// 之后：同时检查 rejected 和 error 字段
let translation = null
if (translationResult.status === 'rejected') {
  console.warn('[Materials API] 翻译数据网络失败:', translationResult.reason)
  warnings.push('Translation data network error')
} else if (translationResult.value.error) {  // ✅ 新增：捕获 Supabase error
  console.warn('[Materials API] 翻译数据查询失败:', translationResult.value.error)
  warnings.push('Translation data query error')
} else {
  translation = translationResult.value.data
}
```

**文件**: [app/api/hospital/materials-page/hospital-info/route.ts:76-106](medical-crm/app/api/hospital/materials-page/hospital-info/route.ts#L76-L106)

### **3. Cache-Control 安全隐患** 🟡
**问题**:
- 使用 `public, s-maxage=60` 会导致共享缓存
- 未来按 `hospitalId` 过滤后可能串数据（不同医院看到对方的数据）

**修复**:
```typescript
// 之前：public（共享缓存）
'Cache-Control': 'public, max-age=60, s-maxage=60'

// 之后：private（仅浏览器缓存）
'Cache-Control': 'private, max-age=60'
```

**文件**: [app/api/hospital/materials-page/hospital-info/route.ts:120](medical-crm/app/api/hospital/materials-page/hospital-info/route.ts#L120)

## ⚠️ **需要认知的行为变化（不是 Bug，但需要知道）**

### **1. `!inner` JOIN 会过滤脏数据**

**影响的 API**:
- `procedures`: `procedures!inner(...)` 会过滤掉 `procedure_id` 指向不存在 procedure 的记录
- `cases`: `procedures!inner(...)` 会过滤掉 `procedure_id` 指向不存在 procedure 的 case

**行为变化**:
- **之前**: 即使数据脏（关联的 procedure 不存在），仍会返回一条记录（procedure 信息缺失）
- **之后**: 脏数据会被 `!inner` JOIN 直接过滤掉，不会出现在结果中

**是否需要修复**:
- ❌ 不需要（这是更合理的行为）
- ✅ 但需要在文档中说明，避免"为什么有些记录突然消失了"的疑问

---

### **2. SWR `keepPreviousData: true` 是冗余的（但无害）**

**大神评价**:
> SWR 本来就不会因为 error 把已有 data 清空（同 key 的场景），所以这项更多是"锦上添花/可能冗余"，但不坏。

**我的回应**: 保留这个配置，作为"防御性编程"，确保在任何情况下都不会丢失已成功的数据。
