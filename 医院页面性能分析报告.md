# 医院页面性能分析报告

## 问题概述

以下医院相关页面加载缓慢：
- `/hospital/messages` - 消息列表和对话详情
- `/hospital/consultations` - 咨询列表
- `/hospital/cases` 和 `/hospital/cases/{id}` - 案例列表和详情
- `/hospital/materials` - 材料管理

---

## 🔴 全局性问题：三个页面都在调用 Cases API

### 问题现象

通过日志发现，**三个不同的页面都在调用 `/api/hospital/cases` API**：

| 页面 | 自己的 API | 额外调用 Cases API |
|-----|-----------|-------------------|
| `/hospital/messages` | `/api/hospital/conversations` 8.2s | `/api/hospital/cases` 7.3s |
| `/hospital/consultations` | `/api/hospital/consultations` 6.3s | `/api/hospital/cases?limit=100` 3.3s |
| `/hospital/cases` | `/api/hospital/cases` 2.2s | - (本身就是) |

### 问题根源猜测

这说明有一个**共用组件**在所有医院页面中调用了 `useCases()`：

```
可能的位置:
├── medical-crm/app/hospital/layout.tsx        <-- 检查这里
├── medical-crm/components/hospital/           <-- 或某个共用组件
└── 某个 Provider 或 Context
```

### 影响分析

- **Messages 页面**: 本身需要 8.2s，额外的 cases API 增加了 7.3s 的数据库负担
- **Consultations 页面**: 本身需要 6.3s，额外加载了 100 个 cases（3.3s）
- 造成**数据库连接竞争**，两个请求同时执行，互相拖慢

### 紧急建议

1. **找到并移除不必要的 `useCases()` 调用**
2. 如果确实需要 cases 数据，考虑：
   - 只在需要的页面加载
   - 减少加载数量（`limit=100` 太多了）
   - 使用缓存避免重复请求

## 1. `/hospital/messages` - 消息页面

### 实际测量结果

```
GET /api/hospital/conversations 200 in 8.2s (首次) / 2.9s (后续)
GET /api/hospital/cases 200 in 7.3s (首次) / 1.8s (后续)
```

**首次加载总耗时: ~8-10 秒**

### 涉及文件
- 页面: `medical-crm/app/hospital/messages/page.tsx`
- API: `medical-crm/app/api/hospital/conversations/route.ts`
- Hooks: `medical-crm/lib/api/hospital/hooks.ts` (第 440-540 行)

### 性能问题

#### 1.0 🔴 **[新发现] 页面同时调用两个 API**

**问题现象**:
Messages 页面不仅调用了 `/conversations` API，还调用了 `/cases` API。

**问题分析**:
- 消息页面为什么需要加载所有 cases？
- 两个 API 同时请求，数据库连接竞争
- 页面需要等待两个 API 都返回才能完全渲染

**需要检查**: `conversation-list.tsx` 中是否有 `useCases()` 调用

#### 1.1 🔴 **[关键问题] Prisma N+1 查询**

**实际 SQL 日志分析**:
```sql
-- 1. 查询 hospitalId
SELECT users.hospital_id FROM users WHERE email = $1

-- 2. 主查询 conversations
SELECT ... FROM conversations LEFT JOIN cases ...

-- 3. 关联查询 cases（Prisma include 导致）
SELECT ... FROM cases WHERE id IN ($1, $2)

-- 4. 关联查询 patient (users)
SELECT ... FROM users WHERE id IN ($1)

-- 5. 关联查询 messages
SELECT ... FROM messages WHERE conversation_id IN ($1,$2,$3,$4,$5)

-- 6. 关联查询 sender (users)
SELECT ... FROM users WHERE id IN ($1,$2)

-- 7. 统计 adminCount
SELECT COUNT(*) FROM conversations WHERE category = 'ADMIN_HOSPITAL' AND hospital_id = $2

-- 8. 统计 patientCount
SELECT COUNT(*) FROM conversations WHERE category = 'HOSPITAL_PATIENT' AND ...
```

**问题分析**:
- 一个 API 请求触发了 **8 次数据库查询**
- Prisma 的 `include` 会把关联拆成多个独立查询，不是 JOIN
- 这是典型的 N+1 问题

**优化建议**:
```typescript
// 方案1: 使用原始 SQL 进行 JOIN 查询
const conversations = await prisma.$queryRaw`
  SELECT c.*, cs.case_number, cs.patient_name, u.name as sender_name
  FROM conversations c
  LEFT JOIN cases cs ON c.case_id = cs.id
  LEFT JOIN messages m ON m.conversation_id = c.id
  LEFT JOIN users u ON m.sender_id = u.id
  WHERE ...
`

// 方案2: 减少 include 层级，只获取必要字段
const conversations = await prisma.conversation.findMany({
  where,
  select: {
    id: true,
    title: true,
    updatedAt: true,
    case: {
      select: { caseNumber: true, patientName: true }
    },
    // 不要 include messages 的 sender，后续按需加载
  }
})
```

#### 1.2 额外的数据库查询计算统计数据

**问题代码** (`conversations/route.ts` 第 161-172 行):
```typescript
// 计算统计 - 按当前医院过滤
const adminCount = await prisma.conversation.count({
  where: {
    category: ConversationCategory.ADMIN_HOSPITAL,
    ...(hospitalId ? { hospitalId } : {}),
  },
})
const patientCount = await prisma.conversation.count({
  where: {
    category: ConversationCategory.HOSPITAL_PATIENT,
    ...(hospitalId ? { case: { assignedHospitalId: hospitalId } } : {}),
  },
})
```

**问题分析**:
- 主查询获取对话列表后，又执行了 **2 个额外的 count 查询**
- 每次刷新页面都会执行这些统计查询

**优化建议**:
```typescript
// 从主查询结果中在内存计算（数据量通常不大）
const adminCount = conversations.filter(c => c.category === 'ADMIN_HOSPITAL').length
const patientCount = conversations.filter(c => c.category === 'HOSPITAL_PATIENT').length
```

#### 1.3 每次请求都查询用户的 hospitalId

**实际 SQL**:
```sql
SELECT users.id, users.hospital_id FROM users WHERE email = $1
```

**问题分析**:
- 每次 API 请求都要查询数据库获取 `hospitalId`
- Keycloak token 中已经有 `hospital_id: "hospital_001"`，但代码没有使用

**优化建议**:
```typescript
// Keycloak payload 中已经有 hospital_id，直接使用
// 但需要注意 token 中是 "hospital_001"，数据库中是 UUID
// 所以需要一个映射表或在 token 中直接存储 UUID
```

#### 1.4 搜索没有防抖 (Debounce)

**问题分析**:
- 用户输入搜索词时，每输入一个字符都会触发 API 请求
- 快速输入 5 个字符 = 5 次 API 请求

**优化建议**:
```typescript
// 使用 useDebouncedValue hook
const debouncedSearch = useDebouncedValue(params.search, 300)

useEffect(() => {
  fetch()
}, [debouncedSearch, params.category])
```

#### 1.5 消息列表没有缓存

**问题分析**:
- 切换对话时，每次都重新获取消息列表
- 没有使用 SWR 或 React Query 等缓存方案
- 相同对话的消息会被重复请求

#### 1.6 🔴 **[新发现] 点击对话触发两个 API，数据重复获取**

**实际测量**:
```
GET /api/hospital/conversations/{id}/messages 200 in 5.3s (6次查询)
GET /api/hospital/conversations/{id} 200 in 6.1s (2次查询)
```

**点击一个对话需要 ~6 秒**

**问题分析**:
```
前端同时调用:
├── useConversationDetail()   -> GET /conversations/{id}
└── useConversationMessages() -> GET /conversations/{id}/messages
```

**重复获取的数据**:
- `messages` 表被查询了 **2 次**
- `users` (sender) 表被查询了 **3 次**
- 总共 **8 次数据库查询**

**messages API 的 6 次查询**:
```sql
1. SELECT conversations WHERE id = $1           -- 获取对话
2. SELECT COUNT(*) FROM messages                -- 消息总数
3. SELECT cases WHERE id IN ($1)                -- 关联 case
4. SELECT messages WHERE conversation_id = $1   -- 获取消息
5. SELECT users WHERE id IN ($1)                -- sender 1
6. SELECT users WHERE id IN ($1,$2)             -- sender 2
```

**detail API 又查询了**:
```sql
1. SELECT messages WHERE conversation_id IN ($1)  -- 重复！
2. SELECT users WHERE id IN ($1)                  -- 重复！
```

**优化建议**:
```typescript
// 方案1: 合并为单一 API
// GET /api/hospital/conversations/{id}/full
{
  conversation: {...},
  messages: [...],
  pagination: {...}
}

// 方案2: detail API 不要 include messages
// messages 由单独的 messages API 获取（避免重复）
const conversation = await prisma.conversation.findUnique({
  where: { id },
  include: {
    case: true,  // 只需要 case 信息
    // 不要 include messages！
  }
})
```

#### 1.7 🟡 编译时间过长

**观察到的现象**:
```
compile: 643ms (messages API)
compile: 646ms (detail API)
```

**问题分析**:
- Next.js 的编译时间占了 ~10% 的响应时间
- Prisma 可能没有使用预编译语句缓存

**优化建议**:
- 确保 Prisma Client 是全局单例（不要每次请求都创建）
- 考虑使用 Prisma Accelerate 或连接池

## 2. `/hospital/consultations` - 咨询列表

### 实际测量结果

```
GET /api/hospital/consultations 200 in 6.3s (首次) / 5.0s (后续)
GET /api/hospital/cases?limit=100 200 in 3.3s (首次) / 1.8s (后续)
```

**首次加载总耗时: ~6.3 秒**

### 涉及文件
- API: `medical-crm/app/api/hospital/consultations/route.ts`
- Hooks: `medical-crm/lib/api/hospital/hooks.ts` (第 272-306 行)

### 性能问题

#### 2.0 🔴 **[确认] 页面同时调用两个 API**

和 messages 页面一样，consultations 页面也调用了 `/cases` API。

**实际日志**:
```
GET /api/hospital/consultations 200 in 6.3s
GET /api/hospital/cases?page=1&limit=100 200 in 3.3s  <-- 为什么？
```

**问题**: 咨询列表页面为什么需要加载 100 个 cases？

#### 2.1 🔴 **[确认] 4 个额外的统计查询**

**实际 SQL 日志**:
```sql
-- 主查询 + 分页
3. SELECT consultations ...
4. SELECT COUNT(*) FROM consultations

-- 关联数据
5. SELECT cases WHERE id IN (...)
6. SELECT users WHERE id IN (...)

-- 4 个统计查询 (我报告里说的全部确认!)
7. SELECT COUNT(*) WHERE status = 'SCHEDULED'
8. SELECT COUNT(*) WHERE status = 'COMPLETED'
9. SELECT COUNT(*) WHERE ai_translation = true AND status = 'SCHEDULED'
10. SELECT COUNT(*) WHERE scheduled_at >= today AND status IN ('SCHEDULED', 'IN_PROGRESS')
```

**问题分析**:
- 一个 API 请求触发了 **10 次数据库查询**
- 其中 4 次是统计查询，可以用单一 SQL 完成

**优化建议**:
```typescript
// 方案1: 使用原始 SQL 聚合查询 (1 次查询代替 4 次)
const stats = await prisma.$queryRaw`
  SELECT
    COUNT(*) FILTER (WHERE status = 'SCHEDULED') as scheduled,
    COUNT(*) FILTER (WHERE status = 'COMPLETED') as completed,
    COUNT(*) FILTER (WHERE scheduled_at >= ${today} AND scheduled_at < ${tomorrow}) as today,
    COUNT(*) FILTER (WHERE ai_translation = true AND status = 'SCHEDULED') as needs_translation
  FROM consultation
`

// 方案2: 从主查询结果在内存计算（如果数据量不大）
const scheduledCount = consultations.filter(c => c.status === 'SCHEDULED').length
const completedCount = consultations.filter(c => c.status === 'COMPLETED').length
// ...
```

#### 2.2 🟡 奇怪的 SELECT 1 查询

**实际 SQL 日志**:
```sql
1. SELECT 1
2. SELECT 1
```

**问题分析**:
- 这两个 `SELECT 1` 是连接池健康检查
- 说明每次请求都在检查连接，可能是连接池配置问题
- 或者 Prisma Client 没有正确复用

#### 2.3 关联数据二次加载

**问题代码** (`consultations/route.ts` 第 46-52 行):
```typescript
// 获取关联的 Case 信息
const caseIds = [...new Set(consultations.map((c) => c.caseId))]
const cases = await prisma.case.findMany({
  where: { id: { in: caseIds } },
  include: { patient: true },
})
```

**问题分析**:
- 先获取 consultations，再根据 caseIds 获取 cases
- 这是一个二次查询，可以通过 include 一次完成

**优化建议**:
```typescript
const consultations = await prisma.consultation.findMany({
  where,
  include: {
    case: {
      include: { patient: true }
    }
  },
  orderBy: { scheduledAt: 'desc' },
})
```

## 3. `/hospital/cases` - 案例列表和详情

### 实际测量结果

```
GET /api/hospital/cases?page=1&limit=20 200 in 2.2s (首次) / 1.8s (后续)
```

**加载时间: ~2 秒**（比 messages 和 consultations 稍快，因为没有额外调用其他 API）

### 涉及文件
- 列表 API: `medical-crm/app/api/hospital/cases/route.ts`
- Hooks: `medical-crm/lib/api/hospital/hooks.ts` (第 83-147 行)

### 性能问题

#### 3.0 🔴 **[确认] 7 次数据库查询，包含不必要的 messages**

**实际 SQL 日志**:
```sql
1. SELECT 1                                          -- 连接检查
2. SELECT COUNT(*) FROM cases                        -- 分页总数
3. SELECT cases ...                                  -- 主查询
4. SELECT users WHERE id IN (...)                    -- 关联 patient
5. SELECT documents WHERE case_id IN (...)           -- 关联 documents
6. SELECT conversations WHERE case_id IN (...)       -- 关联 conversations
7. SELECT messages WHERE conversation_id IN (...)    -- 为什么需要 messages?!
```

**问题分析**:
- 一个案例列表 API 触发了 **7 次数据库查询**
- 第 7 个查询加载了所有对话的消息 - 这是为了计算"未读消息数"
- 但实际上这个计算是**错误的**（见 3.2），所以这些 messages 数据完全是浪费

#### 3.1 🔴 **[确认] 深层嵌套的 include 加载不必要数据**

**问题代码** (`cases/route.ts` 第 70-90 行):
```typescript
const cases = await prisma.case.findMany({
  where,
  include: {
    patient: true,
    documents: {
      where: { status: 'ACTIVE' },
      select: { id: true },
    },
    conversations: {
      include: {
        messages: {        // <-- 加载所有消息
          orderBy: { createdAt: 'desc' },
          take: 1,         // <-- 只需要最新一条
        },
      },
    },
  },
})
```

**问题分析**:
- 为了计算"未读消息数"，加载了所有 `conversations` 和最近的 `messages`
- 对于有多个对话的案例，这会加载大量不必要的数据
- Prisma 的嵌套 include 会生成多个 JOIN 查询

**优化建议**:
```typescript
// 方案1: 使用 _count 聚合
const cases = await prisma.case.findMany({
  where,
  include: {
    patient: true,
    documents: { where: { status: 'ACTIVE' }, select: { id: true } },
    _count: {
      select: { conversations: true }
    }
  },
})

// 方案2: 分开查询未读消息数
const unreadCounts = await prisma.$queryRaw`
  SELECT case_id, COUNT(*) as unread
  FROM message
  WHERE read = false
  GROUP BY case_id
`
```

#### 3.2 在客户端计算未读消息数

**问题代码** (`cases/route.ts` 第 107-110 行):
```typescript
// 计算未读消息数 (简化版本 - 后续可以用实际已读状态)
const unreadMessages = c.conversations.reduce((count, conv) => {
  return count + conv.messages.length
}, 0)
```

**问题分析**:
- 实际上这里的计算是错误的（只计算了最近一条消息的数量，不是未读数）
- 数据处理应该在数据库层完成，减少传输数据量

#### 3.3 🔴 **[确认] 案例详情页多个独立请求 - 问题极其严重**

**实际测量**:
```
GET /api/hospital/cases/{id} 200 in 8.0s (首次) / 2.3s (后续)
GET /api/hospital/cases/{id}/messages 200 in 5.1s (首次) / 928ms (后续)
GET /api/hospital/cases/{id}/documents 200 in 8.4s (首次) / 2.3s (后续)
```

**打开一个案例详情需要 ~8-10 秒！**

**问题分析**:
1. 打开案例详情页触发 **3 个并行 API 请求**
2. 每个 API 还被**重复调用了 2 次**（React Strict Mode 或组件问题）
3. 大量数据被重复查询：

| 表 | 查询次数 | 被哪些 API 查询 |
|----|---------|----------------|
| `conversations` | 4+ 次 | detail, messages, documents 都查了 |
| `messages` | 4+ 次 | detail, messages 都查了 |
| `documents` | 3+ 次 | detail, documents 都查了 |
| `case_progress` | 3+ 次 | detail 查了多次 |

**实际 SQL 日志（部分）**:
```sql
-- /cases/{id} API 查询:
SELECT cases ...
SELECT conversations WHERE case_id = ...
SELECT messages WHERE conversation_id IN ...
SELECT case_progress WHERE case_id = ...
SELECT documents WHERE case_id = ...

-- /cases/{id}/messages API 又查询:
SELECT conversations WHERE case_id = ...    -- 重复！
SELECT messages WHERE conversation_id IN ... -- 重复！
SELECT COUNT(*) FROM messages ...

-- /cases/{id}/documents API 又查询:
SELECT documents WHERE case_id = ...         -- 重复！
```

**优化建议**:
```typescript
// 方案1: 创建一个聚合 API endpoint（推荐）
// GET /api/hospital/cases/{id}/full
const caseDetail = await prisma.case.findUnique({
  where: { id },
  include: {
    documents: true,
    conversations: {
      include: { messages: true }
    },
    progress: true,
  }
})

return {
  caseDetail,
  documents: caseDetail.documents,
  messages: caseDetail.conversations.flatMap(c => c.messages),
}

// 方案2: 至少让各 API 不要重复查询已有数据
// detail API 返回所有数据，其他 API 变成可选
```

#### 3.4 🔴 **[新发现] API 被重复调用**

**问题现象**:
每个 API 都在日志中出现了 **2 次**，说明组件触发了重复请求。

**可能原因**:
1. React Strict Mode 的双重渲染（开发模式）
2. hooks 的依赖数组不正确，导致 effect 重复执行
3. 组件在不同位置被多次使用

**需要检查**:
- `medical-crm/app/hospital/cases/[id]/page.tsx` 中的 hooks 调用
- 确保 `useEffect` 的依赖数组正确

## 4. `/hospital/materials` - 材料管理

### 实际测量结果

```
GET /api/hospital/materials-page/hospital-info 200 in 4.2s (首次) / 1766ms (后续)
```

**加载时间: ~4 秒**

### 涉及文件
- 页面: `medical-crm/app/hospital/materials/page.tsx`
- API: `medical-crm/app/api/hospital/materials-page/hospital-info/route.ts`
- Hooks: `medical-crm/lib/api/hospital/materials-hooks.ts`

### 性能问题

#### 4.0 🟡 **使用 Supabase 而非 Prisma**

**观察**:
日志中没有 `prisma:query`，因为 materials 使用 Supabase 客户端直接查询。

这意味着：
- 无法在终端看到 SQL 日志（需要在 Supabase 后台查看）
- 但从代码分析可知有 **4 个顺序查询**

#### 4.1 🔴 **[确认] API 被重复调用**

**实际日志**:
```
GET /api/hospital/materials-page/hospital-info 200 in 4.2s  (第一次)
GET /api/hospital/materials-page/hospital-info 200 in 1766ms (第二次)
```

和其他页面一样，API 被调用了 **2 次**。

#### 4.2 🔴 **[确认] 顺序执行多个数据库查询（未并行化）**

**问题代码** (`hospital-info/route.ts` 第 15-63 行):
```typescript
// 查询 1: 获取医院
const { data: hospital, error: hospitalError } = await supabase
  .from('hospitals')
  .select('*')
  .single()

// 查询 2: 获取翻译（顺序执行，等待上一个完成）
const { data: translation } = await supabase
  .from('hospital_translations')
  .select('*')
  .single()

// 查询 3: 获取位置
const { data: location } = await supabase
  .from('hospital_location')
  .select('*')
  .single()

// 查询 4: 获取景点
const { data: nearbyAttractions } = await supabase
  .from('hospital_nearby_attractions')
  .select('*')
```

**问题分析**:
- 4 个查询**顺序执行**，假设每个查询 100ms，总耗时 = 400ms
- 这些查询之间没有依赖关系（除了需要 hospital.id）

**优化建议**:
```typescript
// 步骤1: 先获取 hospital.id
const { data: hospital } = await supabase.from('hospitals')...

// 步骤2: 并行获取其他数据
const [translation, location, nearbyAttractions] = await Promise.all([
  supabase.from('hospital_translations').select('*').eq('hospital_id', hospital.id).single(),
  supabase.from('hospital_location').select('*').eq('hospital_id', hospital.id).single(),
  supabase.from('hospital_nearby_attractions').select('*').eq('hospital_id', hospital.id),
])
// 总耗时: 100ms + max(100, 100, 100) = 200ms
```

#### 4.2 页面挂载时同时触发多个 hooks

**问题代码** (`materials/page.tsx` 第 6-30 行):
```typescript
import {
  useHospitalInfo,      // API 请求 1
  useSurgeons,          // API 请求 2
  useProcedures,        // API 请求 3
  useCases,             // API 请求 4
  useProcedureOptions,  // API 请求 5
  // ... 更多 hooks
} from "@/lib/api/hospital/materials-hooks"
```

**问题分析**:
- 页面加载时，**5+ 个 hooks 同时触发 API 请求**
- 没有按需加载（lazy loading）
- 切换 Tab 时没有缓存之前的数据

**优化建议**:
- 使用 React Query 或 SWR 实现缓存
- 按 Tab 懒加载数据
- 合并相关数据为单一 API

## 5. 通用问题

### 5.1 所有 hooks 没有使用缓存

**问题代码** (`hooks.ts` 每个 hook 的模式):
```typescript
export function useSomeData(): UseQueryResult<Data> {
  const [data, setData] = useState<Data | undefined>()
  const [isLoading, setIsLoading] = useState(true)

  useEffect(() => {
    fetch()
  }, [fetch])
  // 每次组件挂载都重新请求
}
```

**问题分析**:
- 没有使用 SWR 或 React Query
- 组件重新挂载会重复请求
- 没有 stale-while-revalidate 模式
- 没有自动重试和错误处理

**优化建议**:
```typescript
// 使用 SWR
import useSWR from 'swr'

export function useCases(params: UseCasesParams) {
  const { data, error, isLoading, mutate } = useSWR(
    ['cases', params],
    () => hospitalApi.GET('/cases', { params: { query: params } }),
    {
      revalidateOnFocus: false,
      dedupingInterval: 5000,
    }
  )
  return { data: data?.data, isLoading, error, refetch: mutate }
}
```

### 5.2 搜索功能没有防抖

几乎所有带搜索的页面都有这个问题：
- Messages 页面
- Cases 页面
- Consultations 页面

### 发现的核心问题

1. **🔴 API 普遍被重复调用 2 次** - 所有页面都有这个问题
   - 可能是 React Strict Mode
   - 或 hooks 依赖数组不正确

2. **🔴 多个页面额外调用 cases API** - messages 和 consultations 都调用了

3. **🔴 Prisma N+1 查询** - include 导致多个独立查询

4. **🔴 数据重复获取** - 多个 API 查询相同的数据

5. **🟡 统计查询未优化** - 4 个 COUNT 可以合并为 1 个

6. **🟡 缺乏客户端缓存** - 没有使用 SWR/React Query
